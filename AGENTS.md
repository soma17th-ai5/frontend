1. 코딩 전에 먼저 생각하기

추측하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 드러내세요.

구현 전에:

자신의 가정을 명확하게 말하세요. 확실하지 않다면 질문하세요.
여러 해석이 가능하다면 조용히 하나를 선택하지 말고, 가능한 해석들을 제시하세요.
더 단순한 접근이 있다면 말하세요. 필요하면 반대 의견도 내세요.
무언가 불명확하다면 멈추세요. 무엇이 헷갈리는지 설명하고 질문하세요. 2. 단순함 우선

문제를 해결하는 최소한의 코드만 작성하세요. 추측성 구현은 하지 마세요.

요청되지 않은 기능은 추가하지 마세요.
한 번만 쓰일 코드를 위해 추상화하지 마세요.
요청되지 않은 “유연성”이나 “설정 가능성”을 추가하지 마세요.
발생할 수 없는 상황을 위한 에러 처리를 만들지 마세요.
200줄 썼는데 50줄로 가능하다면 다시 작성하세요.

스스로에게 물어보세요:

“시니어 엔지니어가 이걸 보고 과하게 복잡하다고 말할까?”

그렇다면 단순화하세요.

3. 외과 수술처럼 최소한만 수정하기

꼭 필요한 부분만 건드리세요. 자신이 만든 흔적만 정리하세요.

기존 코드를 수정할 때:

주변 코드, 주석, 포맷팅을 “개선”하려 하지 마세요.
망가지지 않은 코드를 리팩토링하지 마세요.
본인 스타일이 다르더라도 기존 스타일을 맞추세요.
관련 없는 죽은 코드(dead code)를 발견했다면 언급만 하세요. 삭제하지 마세요.

자신의 변경으로 인해 고아 코드가 생겼다면:

본인의 변경 때문에 사용되지 않게 된 import / 변수 / 함수는 제거하세요.
원래부터 존재하던 죽은 코드는 요청받지 않았다면 제거하지 마세요.

기준:

변경된 모든 줄은 사용자의 요청과 직접 연결되어 있어야 합니다.

4. 목표 중심으로 실행하기

성공 기준을 정의하고, 검증될 때까지 반복하세요.

작업을 검증 가능한 목표로 바꾸세요:

“검증 추가” → “잘못된 입력 테스트를 작성하고 통과시키기”
“버그 수정” → “버그를 재현하는 테스트를 작성한 뒤 통과시키기”
“X 리팩토링” → “변경 전후 테스트가 모두 통과하는지 확인하기”

여러 단계 작업이라면 짧은 계획을 먼저 제시하세요:

[단계] → 검증: [확인 방법]
[단계] → 검증: [확인 방법]
[단계] → 검증: [확인 방법]

강한 성공 기준은 스스로 반복 개선할 수 있게 해줍니다.
약한 기준(예: “작동하게 만들기”)은 계속해서 추가 설명을 요구하게 됩니다.
