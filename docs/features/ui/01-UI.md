# [기능명] AI 챗봇 기본 UI/UX 및 글로벌 레이아웃 세팅

## 1. 개요 (Objective)

- SomaAgent의 메인 인터페이스인 대화형 챗봇 화면의 기본 뼈대(Layout)와 디자인 시스템을 구축한다.
- Figma 디자인(AI-Chatbot-UI-UX-Design)을 기반으로 사용자(User)와 AI(Agent) 간의 채팅 UI 요소들을 원자(Atomic) 단위 컴포넌트로 분리하여 구현한다.

## 2. 기술 스택 (Tech Stack)

- Next.js 16 (App Router)
- Tailwind CSS (스타일링 및 반응형 구현)
- Lucide React (아이콘)
- (선택) Framer Motion (부드러운 메시지 등장 애니메이션 및 UX 개선)

## 3. UI/UX 요구사항 (Requirements)

### 3.1. 글로벌 레이아웃 (Global Layout)

- [ ] **사이드바 (Sidebar):** 좌측에 위치하며, 이전 대화 내역(History)과 새 대화 시작 버튼 포함. 모바일에서는 햄버거 메뉴로 숨김 처리.
- [ ] **메인 영역 (Main Area):** 우측(또는 중앙)에 위치하며, 헤더, 채팅 로그 영역, 하단 입력창으로 구성.

### 3.2. 챗봇 핵심 컴포넌트 (Chat Components)

- [ ] **ChatMessage (말풍선):**
  - **사용자(User):** 우측 정렬, 브랜드 컬러(또는 어두운 색상) 배경.
  - **에이전트(Agent):** 좌측 정렬, 밝은 회색 또는 흰색 배경. SomaAgent 프로필 아이콘 표기.
- [ ] **SourceChip (출처 표시):** AI 답변 하단에 노출되는 뱃지 형태. 클릭 시 해당 링크(OpenSoma 공지 등)로 이동.
- [ ] **ActionButtons (액션 제안 버튼):** AI가 특정 행동(예: "신청하기", "취소하기")을 제안할 때 누를 수 있는 버튼 UI.

### 3.3. 입력창 (Chat Input Area)

- [ ] **Textarea:** 사용자가 길게 입력할 수 있도록 글자 수에 따라 높이가 자동 조절(Auto-resize)되는 텍스트 영역.
- [ ] **버튼:** 전송(Send) 버튼 및 첨부파일/도구(툴) 호출 아이콘 배치.
- [ ] 빈 값일 때는 전송 버튼 비활성화 (Disabled 상태 처리).

### 3.4. UX 인터랙션 (UX Interactions)

- [ ] **Scroll to Bottom:** 새로운 메시지가 추가되거나 AI가 답변을 생성 중일 때, 자동으로 스크롤이 가장 아래로 부드럽게 이동(Smooth Scroll)해야 함.
- [ ] **Loading State:** AI가 답변을 생성하는 동안 사용자에게 진행 중임을 알리는 로딩 인디케이터(예: 타이핑 애니메이션 쩜쩜쩜 `...` 또는 스켈레톤 UI) 노출.

## 4. 폴더 구조 가이드 (Component Structure)

## 6. 구현 과정 (Implementation Log)

### 6.1. 사전 정리

- 본 프로젝트는 Next.js 16 + App Router 기본 구조를 사용하고 있어 `src/`를 두지 않는다. 따라서 가이드의 `src/components/...`는 동일한 의미로 **루트의 `components/`** 에 매핑했다.
- `tsconfig.paths`가 `@/*` → `./*`로 설정되어 있으므로 모든 import는 `@/components/...`, `@/lib/...`, `@/hooks/...` 형태로 통일했다.

### 6.2. 의존성 추가

- 스펙에 명시된 **Lucide React** 아이콘을 위해 `lucide-react`를 설치했다.
  ```bash
  npm install lucide-react
  ```
- Framer Motion은 (선택) 항목이므로 1차 구현에서는 도입하지 않고, 추후 메시지 등장 애니메이션이 필요한 시점에 추가하기로 했다.

### 6.3. 글로벌 디자인 토큰 정리

- `app/globals.css`에 Tailwind v4의 `@theme inline` 기반으로 **브랜드 컬러 토큰**(`--primary`, `--surface`, `--border`, `--muted-foreground` 등)을 선언했다.
- 한국어 가독성을 위해 `Apple SD Gothic Neo`, `Noto Sans KR`을 fallback에 추가하고, 채팅 영역 전용 스크롤바 스타일(`.chat-scroll`)과 textarea 자동 높이용 `field-sizing: content`를 함께 정의했다.
- `app/layout.tsx`의 `<html lang>`을 `ko`로, 메타데이터를 `SomaAgent · AI 생활 비서`로 교체했다.

### 6.4. 라우팅 구성

- `app/page.tsx` → `/login`으로 `redirect()` 처리.
- `app/login/page.tsx` → 첫 진입용 로그인 화면.
- `app/chat/page.tsx` → 메인 채팅 화면.
- 로그인 폼의 메인 CTA를 `<Link href="/chat">`로 연결해 임시 UI 흐름에서도 클릭으로 채팅 화면까지 이동 가능하게 했다.

### 6.5. 컴포넌트 분해 (Atomic 단위)

스펙의 4번 폴더 구조 가이드를 그대로 따라 다음과 같이 분리했다.

| 레이어 | 파일 | 역할 |
| ------ | ---- | ---- |
| layout | `components/layout/Sidebar.tsx` | 새 대화 / 검색 / 최근 대화 / 사용자 푸터. lg 이상에서는 정적, 모바일에서는 오버레이 슬라이드. |
| layout | `components/layout/ChatHeader.tsx` | 좌측 햄버거(모바일) + 브랜드 타이틀 + 우측 프로필. |
| chat   | `components/chat/ChatContainer.tsx` | 메시지 리스트 스크롤 컨테이너. `useScrollToBottom`을 연결. |
| chat   | `components/chat/ChatMessage.tsx` | 역할(`user` / `agent` / `system`)에 따라 렌더링 분기. |
| chat   | `components/chat/ChatInput.tsx` | 자동 높이 textarea + 첨부/음성/전송 버튼 + Disabled 처리. |
| chat   | `components/chat/MentoringCard.tsx` | 멘토링 카드(신청/마감 상태 분기). |
| chat   | `components/chat/NoticeBlock.tsx` | 공지 요약 bullet 블록. |
| chat   | `components/chat/WebexThread.tsx` | Webex 인용 메시지 리스트. |
| chat   | `components/chat/StatusToast.tsx` | "신청 완료" 같은 시스템 알림 토스트. |
| ui     | `components/ui/SourceChip.tsx` | 출처 뱃지(`neutral` / `warm` 톤, 외부 링크 지원). |
| ui     | `components/ui/ActionButton.tsx` | 액션 제안 버튼(`primary` / `ghost` 변형). |

요구사항 3.2~3.3 항목은 위 컴포넌트 단위로 1:1 매핑된다.

### 6.6. 데이터/상태 분리

- 임시 데이터는 `lib/mockChat.ts`에 모아 두고, 메시지 타입을 `role`/`kind`로 **discriminated union**으로 정의했다. 이렇게 하면 새 메시지 종류(예: 차트, 파일 카드)가 추가될 때 `ChatMessage` 분기에서 타입이 강제된다.
- 사이드바의 최근 대화 리스트도 같은 모듈의 `MOCK_HISTORY`로 분리해, 추후 API 응답으로 교체할 때 컴포넌트 수정이 필요 없게 했다.

### 6.7. UX 훅 구현

- `hooks/useScrollToBottom.ts`에 제네릭 훅을 작성했다. 인자로 받은 의존성 배열이 바뀔 때마다 컨테이너를 `behavior: "smooth"`로 맨 아래까지 스크롤한다.
- `ChatContainer`에서 `messages.length`를 의존성으로 전달해 새 메시지 추가 시 자동 스크롤되도록 연결했다.
- 로딩 인디케이터(요구사항 3.4의 두 번째 항목)는 컴포넌트 자리만 비워 두고, 실제 스트리밍 응답이 붙는 단계에서 `ChatMessage` 안의 `kind: "text"` 분기에 typing dots 컴포넌트를 끼우기로 했다.

### 6.8. 입력창 처리 (요구사항 3.3)

- `ChatInput.tsx`에서 textarea의 높이를 두 가지 방식으로 동시에 보장:
  1. CSS `field-sizing: content` (최신 브라우저)
  2. `useRef` + `scrollHeight` 기반 폴백 (구형 브라우저 대응)
- `value.trim().length === 0`일 때 전송 버튼을 `disabled` 처리하고, Tailwind의 `disabled:` 변형으로 시각적으로도 비활성 톤을 입혔다.
- `Enter` → 전송, `Shift + Enter` → 줄바꿈 단축키를 함께 구현했다.

### 6.9. 반응형 (Mobile-first)

- Sidebar는 기본 `-translate-x-full`로 숨겨 두고 모바일에서 햄버거로 토글하며, `lg` 이상에서 `lg:translate-x-0 lg:static`으로 항상 노출.
- 로그인 페이지는 `grid-cols-1 lg:grid-cols-[1.05fr_1fr]`로 좌측 브랜드 패널을 데스크톱에서만 노출하고, 모바일에서는 상단에 축약 로고가 보이도록 처리.
- 채팅 본문 폭은 `max-w-3xl`로 제한하고 좌우 패딩을 `px-4 sm:px-6`으로 단계 조정.

### 6.10. 검증

- `npx tsc --noEmit`, `npx eslint app components hooks lib` 모두 에러 없이 통과.
- `npm run dev`로 `/login` → `/chat` 이동 흐름과 첨부 이미지의 대화 시퀀스(멘토링 → 신청 완료 → 공지 요약 → Webex 검색)를 시각적으로 확인.

## 7. 코드 리뷰 피드백 반영 (Review Follow-up)

`docs/features/01-UI-Feedback.md`(Gemini 코드 리뷰)에 정리된 항목을 다음과 같이 처리했다.

### 7.1. 반영한 항목

#### (1) `app/chat/page.tsx` — 임시 안내 토스트 제거

- **피드백:** "임시 UI라 실제로는 동작하지 않습니다" 문구가 사용자에게 노출되고 있어 UX First 원칙에 어긋남.
- **수정 이유:** 도움말 기능이 실제로 연결되지 않은 상태에서 "임시"임을 사용자에게 설명하는 UI는 신뢰도를 떨어뜨린다. _제거하거나 실제 기능으로 대체_ 가운데 현시점에서는 도움말 모달이 없으므로 **제거**가 맞다고 판단했다.
- **수정 과정:**
  - `showHint` 로컬 상태와 검정 토스트 마크업, `X`(close) 아이콘 import 모두 삭제.
  - 도움말 진입점인 `CircleHelp` FAB는 그대로 두어 추후 도움말 모달이 붙을 자리만 보존.
  - 결과적으로 `ChatPage`의 책임이 "사이드바·헤더·메시지·입력창의 셸 조립"으로 더 깔끔해졌다.

#### (2) `app/login/page.tsx` — `GoogleIcon` 컴포넌트 분리

- **피드백:** `LoginPage` 안에 Google SVG가 인라인으로 들어가 있어 단일 책임 원칙을 위반함.
- **수정 이유:** 로그인 페이지의 책임은 _레이아웃 + 폼 로직_ 이고, 아이콘은 다른 화면(예: 사용자 설정, 외부 계정 연동)에서 재사용 가능성이 충분하다.
- **수정 과정:**
  - 신규 파일 `components/ui/GoogleIcon.tsx` 생성. `Props = { className?: string }` 만 노출.
  - 기존 단색 SVG를 **공식 Google 컬러 4색** 마크로 교체해 브랜드 톤을 정확히 따름.
  - `LoginPage`에서 인라인 함수 컴포넌트를 삭제하고 `import { GoogleIcon } from "@/components/ui/GoogleIcon"` 으로 대체.

#### (3) `app/login/page.tsx` — 제출 시 로딩/에러 UI 패턴 스캐폴딩

- **피드백:** 실제 인증 로직, 로딩, 실패 메시지가 누락되어 있어 UX First / Error Handling 관점에서 개선 필요.
- **수정 이유:** 백엔드 API는 아직 없지만, **API 연결 전에 "UX 패턴" 자체를 미리 잡아 두면** 추후 fetch 호출만 끼워 넣어도 디자인이 깨지지 않는다. 패턴을 비워 두면 늘 마지막 순간에 급하게 붙이게 된다.
- **수정 과정:**
  - `useRouter`, `isSubmitting`, `error` 상태 도입.
  - `<Link href="/chat">`을 `<button type="submit">`으로 바꾸고 `<form onSubmit={handleSubmit}>`에서 일괄 처리.
  - 빈 입력 시 인라인 빨간 배너(`role="alert"` + `TriangleAlert` 아이콘) 노출 → 추후 서버 응답 메시지로 그대로 교체 가능.
  - 제출 중에는 입력 필드와 모든 버튼을 `disabled` 처리하고, 메인 CTA에 `Loader2` 스피너 + "로그인 중…" 라벨 노출.
  - 실제 API 자리에는 `// TODO(api):` 주석으로 교체 지점을 표시하고, 임시로 `setTimeout(600ms)`만 두어 로딩 UX를 시연.

### 7.2. 보류한 항목 (의도적 미반영)

#### (1) `ChatContainer` — React Query / SWR 도입

- **피드백:** 향후 API 연동을 대비한 서버 상태 관리 라이브러리 도입 권장.
- **보류 이유:** 현재 `ChatContainer`는 **순수하게 props로 받은 `messages` 배열을 렌더링하기만 한다.** 이 단계에서 React Query/SWR을 끼우는 것은 다음과 같은 이유로 시기상조라고 판단했다.
  1. **데이터 출처가 결정되지 않음:** 채팅 메시지는 일반 GET이 아니라 **서버 액션 + 스트리밍 응답**이 될 가능성이 높다. 그 경우 React Query보다 `useChat` 류의 전용 훅이나 SSE/WebSocket 추상화가 더 적합하다.
  2. **YAGNI:** 추상화를 미리 도입하면 mock 데이터 한 군데를 위해 캐시 키·queryFn·mutation을 설계하게 되고, 추후 실제 통신 모델이 정해지면 다시 갈아엎어야 한다.
  3. **대안 표시:** 후속 과제 6.11에 `ChatInput.onSend ↔ messages 상태 연결` 항목을 그대로 두어, 통신 모델이 확정될 때 함께 결정하도록 트래킹.

#### (2) `ChatMessage` — `kindToComponentMap` 패턴 도입

- **피드백:** `kind`별 분기가 늘어날 경우 컴포넌트 매핑 객체로 분리.
- **보류 이유:** 리뷰에서도 "현재 코드는 허용 가능한 수준"이라고 명시. 현재 `ChatMessage`는 다음 두 가지 이유로 **명시적 switch 형태가 더 안전**하다고 판단했다.
  1. **Discriminated union 좁히기:** `kind`마다 props 형태가 다르므로(`mentoring.cards`, `notice.items`, `webex.replies` …), `Map<kind, Component>` 형태는 props를 `any`로 풀거나 별도 어댑터를 두지 않으면 타입 안전성이 떨어진다.
  2. **공통 UI 셸 공유:** 모든 agent 메시지가 `프로필 + 흰색 카드 + SourceChip` 셸을 공유한다. 이 셸을 dispatcher 안에서 한 번에 감싸는 현재 방식이 가장 중복이 적다.
  3. **트리거 조건:** `kind`가 한두 개 더 늘어나는 시점에 다시 검토. (트래킹용 항목으로 6.11에 별도 추가 예정 없음 — 임계점 도달 시 리팩터링.)
